4 transaction isolation levels

There are 4 transaction isolation levels:

  1. Read uncommitted
  2. Read commited
  3. Repeatable read
  4. Serializable
There have been created in order to privide date integrity. When databse ist used by 2 or more users: user nr1 only is reading those data but in the same time user nr2 is makeing updates on those deta. So there are crucial questions:
  • How can we manage concurrent access to a database properly?
  • How are we able to handle multiple transactions?
And what is crucial date have to stay consistent. And that way datebases provide transaction isolation levels. What that mean is shortly describe in the table below which I encourage you to read.
Read uncommitted: Transaction nr1 is reading modified data by transaction nr2 before the transaction nr2 is commited.
Read commited: Because of the transaction nr.1 is locking the transaction nr2 cannot read date aslong as the transaction nr1 is not ended. This is a solution for the problem "dirty read" - that is reading the data which have not been neither commited nor rollbacked.
Repeatable read: Transakcja nr.1 is locking while the data are reading. And this is untill the transaction nr1 is ended. In practical way this mean that another transactions cannot make any other modifications.
Serializable: Transaction nr.1 is locking untill it's end. Another transactions cannot make any: changes and nither added nor deleteed date. The other transaction are waitng when the transaction nr1 will be finished.
