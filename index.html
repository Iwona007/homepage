4 transaction isolation levels

There are 4 transaction isolation levels:

  1. Read uncommitted
  2. Read commited
  3. Repeatable read
  4. Serializable
There have been created in order to provide date integrity. When database is used by 2 or more users: user nr1 is only reading the data but in the same time user nr2 is making updates on that data. So, there are crucial questions:
  • How can we manage concurrent access to a database properly?
  • How are we able to handle multiple transactions?
And what is crucial is that the data has to stay consistent. That way databases provide transaction isolation levels. What that means is, shortly described in the table above, which I encourage you to read.
Read uncommitted: Transaction nr1 is reading modified data by transaction nr2 before the transaction nr2 is committed.
Read commited: Because of the transaction nr.1 is locking the transaction so that transaction nr2 cannot read dataas long as the transaction nr1 is not ended. This is a solution for the problem "dirty read" - that is reading the datas which have not been neither committed nor rollbacked.
Repeatable read: Transaction nr.1 is locking while the data are reading. And this is until the transaction nr1 has ended. In a practical way this means that other transactions cannot make any other modifications.
Serializable: Transaction nr.1 is locked until it's end. Other transactions cannot make any: changes and neither added nor deleted date. The other transactions are waiting when the transaction nr1 will be finished.
